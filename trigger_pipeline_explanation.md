# Упражнение на чтение .yml конфигураций. Часть 1

Разберем YAML‑конфиг GitHub Actions из работы https://github.com/softboxdev/devops/blob/dev/deploy.md пошагово — с объяснением каждой секции и деталей работы пайплайна.

## Общая структура

Конфиг описывает пайплайн CI/CD для Node.js‑приложения: сначала выполняются тесты, затем (при успехе и на нужных ветках) — деплой на VPS.

## Разбор секций

### 1. Метаданные пайплайна

```yaml
name: Deploy to VPS
```
* `name` — название пайплайна, отображается в интерфейсе GitHub Actions.

### 2. Триггеры запуска (`on`)

```yaml
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
```

Определяет, когда запускать пайплайн:
* `push` на ветки `main` или `master` — запускает пайплайн при каждом пуше в эти ветки.
* `pull_request` в ветки `main` или `master` — запускает пайплайн для всех PR, нацеленных на эти ветки (позволяет проверять код до мерджа).

### 3. Задание «test» (`jobs.test`)

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
```

* `jobs` — блок, содержащий все задания пайплайна.
* `test` — название задания для запуска тестов.
* `runs-on: ubuntu-latest` — задание выполняется на виртуальном окружении Ubuntu последней версии от GitHub Actions.

#### Шаги задания `test`:

1. **Checkout кода:**
   ```yaml
   - uses: actions/checkout@v4
   ```
   * Клонирует репозиторий в рабочее окружение. Без этого шага дальнейшие шаги не будут иметь доступа к коду.

2. **Настройка Node.js:**
   ```yaml
   - name: Setup Node.js
     uses: actions/setup-node@v4
     with:
       node-version: '18'
   ```
   * Устанавливает указанную версию Node.js (`18`) в окружении.
   * `uses` — использование готового действия из GitHub Marketplace.
   * `with` — передача параметров действию.

3. **Установка зависимостей:**
   ```yaml
   - name: Install dependencies
     run: npm install
   ```
   * Выполняет команду `npm install` для установки зависимостей из `package.json`.

4. **Запуск тестов:**
   ```yaml
   - name: Run tests
     run: npm test
   ```
   * Запускает тесты, определённые в скрипте `test` файла `package.json`. Если тесты падают, пайплайн прерывается.

### 4. Задание «deploy» (`jobs.deploy`)

```yaml
deploy:
  runs-on: ubuntu-latest
  needs: test
  if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/master'
```

* `deploy` — название задания деплоя.
* `runs-on: ubuntu-latest` — выполняется на Ubuntu.
* `needs: test` — задание запускается только после успешного выполнения задания `test`. Если тесты не прошли, деплой не начнётся.
* `if` — условие выполнения задания. Деплой запускается только при пуше в ветки `main` или `master`. Для PR это задание не выполняется.

#### Шаги задания `deploy`:

1. **Checkout кода (повторный):**
   ```yaml
   - name: Checkout code
     uses: actions/checkout@v4
   ```
   * Повторно клонирует код — необходимо, так как задание `deploy` запускается в новом окружении.

2. **Настройка SSH‑агента:**
   ```yaml
   - name: Setup SSH
     uses: webfactory/ssh-agent@v0.8.0
     with:
       ssh-private-key: ${{ secrets.SSH_PRIVATE_KEY }}
   ```
   * Подготавливает SSH‑ключ для аутентификации на сервере.
   * Использует стороннее действие `webfactory/ssh-agent`.
   * `secrets.SSH_PRIVATE_KEY` — приватный ключ берётся из секретов репозитория GitHub (не отображается в логах).

3. **Добавление сервера в known_hosts:**
   ```yaml
   - name: Add server to known_hosts
     run: |
       mkdir -p ~/.ssh
       ssh-keyscan -H ${{ secrets.SERVER_IP }} >> ~/.ssh/known_hosts
   ```
   * Добавляет отпечаток сервера в файл `known_hosts`, чтобы SSH‑соединение не запрашивало подтверждения.
   * `ssh-keyscan` — утилита для получения публичного ключа сервера.
   * `${{ secrets.SERVER_IP }}` — IP‑адрес сервера берётся из секретов.

4. **Деплой на сервер:**
   ```yaml
   - name: Deploy to server
     run: |
       ssh ${{ secrets.SSH_USERNAME }}@${{ secrets.SERVER_IP }} "
         # Создаем папку для проекта
         mkdir -p ~/apps/my-app
         
         # Копируем файлы проекта
         rsync -avz --delete ./ ${{ secrets.SSH_USERNAME }}@${{ secrets.SERVER_IP }}:~/apps/my-app/
         
         # Заходим в папку и запускаем деплой скрипт
         cd ~/apps/my-app
         chmod +x deploy.sh
         ./deploy.sh
       "
   ```

   **Что происходит внутри команды `run`:**
   1. Устанавливается SSH‑соединение с сервером (`ssh username@ip`).
   2. На сервере создаётся директория для проекта (`mkdir -p`).
   3. Файлы синхронизируются с помощью `rsync`:
      * `-a` — архивный режим (сохраняет права, владельца и т.д.).
      * `-v` — подробный вывод.
      * `-z` — сжатие при передаче.
      * `--delete` — удаляет файлы на сервере, которых нет локально.
   4. На сервере переходят в папку проекта, делают скрипт `deploy.sh` исполняемым (`chmod +x`) и запускают его.

## Ключевые механизмы безопасности

* **Секреты (`secrets`):** все чувствительные данные (SSH‑ключ, IP, логин) хранятся в настройках репозитория GitHub как секреты. Они не отображаются в логах и недоступны для чтения в интерфейсе.
* **Условный деплой:** деплой выполняется только для веток `main`/`master` и только после успешного прохождения тестов. Это предотвращает случайный деплой нерабочего кода.
* **SSH‑аутентификация:** используется приватный ключ вместо пароля, что безопаснее.

## Поток выполнения пайплайна

1. Кто‑то делает `git push` в ветку `main` или открывает PR в `main`.
2. Запускается пайплайн.
3. Выполняется задание `test`: код клонируется, устанавливается Node.js 18, ставятся зависимости, запускаются тесты.
4. Если тесты прошли:
   * Для PR пайплайн завершается (деплой не запускается из‑за условия `if`).
   * Для пуша в `main`/`master` запускается задание `deploy`.
5. В задании `deploy`: код клонируется снова, настраивается SSH, сервер добавляется в `known_hosts`.
6. Через SSH выполняется набор команд на сервере: создание папки, синхронизация файлов, запуск скрипта `deploy.sh`.

---

## Что должен содержать скрипт `deploy.sh`

Скрипт `deploy.sh`, запускаемый на сервере, обычно выполняет:
* остановку старого экземпляра приложения (если есть);
* установку зависимостей (`npm install`);
* сборку (если нужно, `npm run build`);
* запуск приложения (например, через PM2, systemd или напрямую `npm start`);
* перезапуск сервисов (nginx, если нужно).

# Упражнение на чтение .yml конфигураций. Часть 2

Разберем переменные из конфига GitHub Actions на **постоянные (стандартные)** и **изменяемые (зависящие от проекта/ОС)**.

## Постоянные переменные (стандартные)

Это встроенные переменные среды GitHub Actions — они всегда доступны, не требуют настройки и работают одинаково для всех проектов.

**Основные примеры в данном конфиге:**

* `github.ref` — полный путь к текущей ветке или тегу (например, `refs/heads/main`). Используется в условии `if` для запуска деплоя только на определённых ветках.
* `github.repository` — имя репозитория в формате `owner/repo` (например, `myuser/myproject`).
* `github.actor` — имя пользователя, который запустил пайплайн.
* `github.sha` — хеш коммита, для которого запущен пайплайн.
* `github.event_name` — тип события, запустившего пайплайн (`push`, `pull_request` и т. д.).

**Особенности:**
* Предоставляются GitHub автоматически.
* Не нужно объявлять или настраивать.
* Имеют фиксированный формат и значение.
* Используются для условной логики, идентификации окружения и метаданных.

## Изменяемые переменные (зависящие от проекта и ОС)

Эти значения уникальны для каждого проекта и окружения. Их нужно настраивать отдельно.

### Группа 1. Зависимые от проекта

Определяются структурой и требованиями конкретного проекта:

* **Версии ПО:**
   * `node-version: '18'` — версия Node.js. Может быть `16`, `20` и т. д. в зависимости от требований проекта.
   * Версии действий (`actions/checkout@v4`, `actions/setup-node@v4`) — могут меняться при обновлении действий.
* **Ветки для триггеров:**
   ```yaml
   branches: [ main, master ]
   ```
   Могут быть `develop`, `production` и т. д. — зависит от стратегии ветвления проекта.
* **Команды сборки и тестов:**
   * `npm install` — может быть `yarn install`, `pnpm install`.
   * `npm test` — может быть `npm run test:unit`, `jest`, `mocha` и т. д.
* **Структура проекта:**
   * Пути в `rsync` (`~/apps/my-app/`) — зависят от соглашения по развёртыванию на сервере.
   * Имя и расположение скрипта деплоя (`deploy.sh`) — может быть `deploy.sh`, `deploy.py`, `release.sh` и т. д.
* **Название пайплайна:**
   * `name: Deploy to VPS` — может быть любым (`CI/CD Pipeline`, `Build and Test` и т. д.).

### Группа 2. Зависимые от операционной системы (ОС)

Значения, которые меняются в зависимости от ОС сервера или runner’а:

* **Пути файловой системы:**
   * В Linux/macOS: `~/apps/my-app` (домашняя директория).
   * В Windows: `C:\projects\my-app` — требует другого синтаксиса.
* **Командная оболочка и утилиты:**
   * Linux/macOS: используют `bash`, `rsync`, `ssh`, `chmod`.
   * Windows: используют `PowerShell`, `robocopy`, `schtasks` и т. д. Команды `rsync` и `ssh` могут отсутствовать или требовать установки.
* **Права доступа:**
   * Команда `chmod +x deploy.sh` работает в Unix‑системах.
   * В Windows управление правами доступа выполняется иначе.
* **Синтаксис команд:**
   * Разделители путей: `/` в Linux vs `\` в Windows.
   * Команды для создания директорий: `mkdir -p` в Linux vs `mkdir` или `New-Item` в Windows.
* **Окружение выполнения:**
   * `runs-on: ubuntu-latest` — можно заменить на `windows-latest` или `macos-latest`, что повлияет на доступные команды и утилиты.

### Группа 3. Зависимые от инфраструктуры (серверов и секретов)

Уникальные для каждого развёртывания:

* **Данные для доступа к серверу:**
   * `${{ secrets.SERVER_IP }}` — IP‑адрес целевого сервера.
   * `${{ secrets.SSH_USERNAME }}` — пользователь для подключения по SSH.
   * `${{ secrets.SSH_PRIVATE_KEY }}` — приватный SSH‑ключ для аутентификации.
* **Порты и протоколы:**
   * Порт для SSH (по умолчанию 22) может быть изменён.
   * Могут использоваться другие протоколы (SFTP, FTP) вместо SSH/rsync.
* **Конфигурация сети:**
   * Настройки firewall, VPN, прокси — влияют на возможность подключения к серверу.
* **Окружение на сервере:**
   * Установленное ПО (Node.js, Python, Docker и т. д.) — должно соответствовать требованиям проекта.
   * Наличие скриптов (`deploy.sh`) и их функционал.

## Важные замечания

1. **Секреты (`secrets`)** всегда относятся к изменяемым данным — они уникальны для каждого репозитория и окружения. Никогда не хранятся в коде, а задаются в настройках репозитория.
2. **Условные операторы** (`if: github.ref == ...`) используют постоянные переменные (`github.ref`), но сами условия зависят от проекта (какие ветки считать «основными»).
3. **Версии действий** (`@v4`) можно зафиксировать для стабильности или использовать `@latest` для автоматических обновлений — это тоже решение на уровне проекта.
