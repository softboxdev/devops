## 1. DevOps и ITSM (IT Service Management)

### Суть пересечения
ITSM — это набор процессов для управления IT-услугами на протяжении их жизненного цикла (классика — ITIL). DevOps раньше воспринимался как антипод ITSM из-за акцента на скорость, а не на контроль. Однако на практике они дополняют друг друга.

### Где ITSM помогает DevOps
- **Управление инцидентами** — автоматизированные реакции на сбои в CI/CD.
- **Управление изменениями** — вместо тяжёлых Change Advisory Board (CAB) используются «низкорисковые стандартные изменения» с автоматическим одобрением.
- **Управление конфигурациями** — всё как код (IaC), что идеально вписывается в CMDB.
- **Service Desk** — интеграция чат-ботов и автоматических уведомлений о деплоях.

### Где DevOps улучшает ITSM
- Ускорение согласования изменений через Git + автоматические политики.
- Внедрение «Change as Code» — запрос на изменение = Pull Request.
- Непрерывное улучшение процессов (по Lean/Agile) внутри ITSM.

### Итог по паре
> *DevOps не отменяет ITSM, а трансформирует его в более гибкую, автоматизированную и менее бюрократическую форму.*

## 2. Бережливые практики (Lean)

### Основные принципы Lean (по Д. Уомаку и Т. Джонсу)
1. Ценность для клиента.
2. Карта потока создания ценности (Value Stream Mapping).
3. Обеспечение непрерывного потока.
4. Вытягивающее производство (pull).
5. Стремление к совершенству.

### Как Lean применяется в DevOps

| Lean-концепция | Воплощение в DevOps |
|----------------|----------------------|
| Устранение потерь (муда) | Лишние согласования, ручные деплои, ожидание тестов, переходы между командами |
| Бережливое производство | Ограничение незавершённой работы (WIP) через Kanban в DevOps-командах |
| Стандартизация работы | Pipeline as Code, единые скрипты сборки, унифицированные окружения |
| Кайдзен (непрерывное улучшение) | Ретроспективы + метрики (DORA: lead time, deployment frequency, MTTR, change fail rate) |

### Ценностная карта (Value Stream Map) для DevOps
Пример: от коммита до продакшена. Находят узкие места:
- Ручное тестирование (7 дней) → автоматизировать.
- Ожидание очереди развёртывания → внедрить CI/CD.

## 3. Использование Agile

### Agile как культурная основа
DevOps часто называют «расширением Agile на операции» (Ops). Agile декларирует:
- Быструю обратную связь.
- Адаптивность.
- Командную ответственность.

### Конкретные Agile-ритуалы в DevOps
- **Ежедневные стендапы** — синхронизация разработки и эксплуатации.
- **Планирование итераций** — включение Ops-задач (патчи, масштабирование) в бэклог.
- **Ретроспективы** — улучшение пайплайнов и взаимодействия Dev+Ops.
- **Демо** — показываются работающие автоматизированные сценарии.

### Отличие DevOps от Agile
Agile в основном про разработку (команды Dev), а DevOps включает:
- Инфраструктуру.
- Мониторинг.
- Безопасность (DevSecOps).
- Релизные процессы в production.

Тем не менее без Agile DevOps превращается в набор инструментов, а не в культуру.

## 4. Scrum как основа DevOps

### Почему Scrum часто берут за основу
Scrum даёт чёткие роли, артефакты и события. Для DevOps это полезно:
- Прозрачность работы и над кодом, и над инфраструктурой.
- Спринты (1–2 недели) хорошо ложатся на частые релизы.
- Определение «Done» включает деплой в production или staging с мониторингом.

### Как адаптируют Scrum под DevOps

| Scrum-элемент | Адаптация для DevOps |
|----------------|----------------------|
| Product Owner | Учитывает Ops-требования (надёжность, производительность) как равноправные |
| Scrum-мастер | Помогает убрать барьеры между Dev и Ops |
| Команда | Кросс-функциональная: разработчики, тестировщики, SRE, админы |
| Бэклог спринта | Содержит: фичи, техдолг, автоматизацию тестов, обновление мониторинга |
| Спринт-ревью | Демонстрация работы пайплайна и метрик (например, время восстановления) |
| Ретроспектива | Анализ сбоев деплоя, узких мест в CI/CD |

### Пример успешного симбиоза
Команда использует Scrum-доску, где есть колонки:
- To Do → In Dev → In Test → In Staging → In Prod (Done).
На каждом этапе — автоматические проверки (Jenkins, GitLab CI).
В конце спринта — реально выкаченный в production инкремент.

## 5. Общая схема организационных источников DevOps

```
       Lean (устранение потерь, поток ценности)
                |
                v
Agile (быстрая обратная связь, адаптивность) <---> Scrum (кросс-функц. команда, спринты)
                |
                v
            DevOps
                |
                v
     ITSM (управление инцидентами, изменениями, конфигурациями)
```

**Ключевой вывод:** DevOps не изобретает всё с нуля, а заимствует и синтезирует:
- из **Lean** — мышление о ценности и потоке,
- из **Agile** — итеративность и коллаборацию,
- из **Scrum** — ритм и прозрачность,
- из **ITSM** — дисциплину управления IT-услугами.

## 6. Вопрос к аудитории

> **Вопрос:** Объясните, почему внедрение DevOps не отменяет процесс управления изменениями из ITSM, а трансформирует его. Приведите пример.

**Краткий ответ:**  
DevOps автоматизирует управление изменениями через GitOps (Pull Request → автоматическое развёртывание в staging → метрики → автопромоушн в prod). Классический Change Advisory Board заменяется автоматическими политиками для низкорисковых изменений, но для критических изменений может сохраняться ручное утверждение. Так процесс становится быстрее, но не исчезает.
