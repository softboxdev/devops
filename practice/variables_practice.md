### Практическое задание: работа с переменными в GitHub Actions 


**Цель:** научиться различать и использовать постоянные (встроенные) и изменяемые переменные в GitHub Actions, адаптировать пайплайн под свой проект и ОС. Обратите внимание: что задание не на деплой приложения, а на умение читать конфигурации. Задание не предполагает практического выполнения всех шагов, шаги приведены для демонстрации. 
Задание состоит в том, что необходимо ответить на Контрольные вопросы.

**Время выполнения:** 60–90 минут.

**Требования:**
* Аккаунт на GitHub.
* Репозиторий с простым Node.js‑приложением (можно использовать любой тестовый проект).
* Доступ к VPS или локальному серверу с Linux (Ubuntu/Debian) для деплоя.
* Базовые знания Git, Node.js и командной строки Linux.

#### Шаг 1. Подготовка репозитория

1. Создайте новый репозиторий на GitHub или используйте существующий.
2. Загрузите в него простое Node.js‑приложение с:
   * `package.json` (с скриптами `test` и `start`);
   * базовым тестом (например, один Jest‑тест);
   * файлом `deploy.sh` со следующим содержимым:
```bash
#!/!/bin/bash
echo "Starting deployment..."
npm install
npm run build  # если есть сборка
pm2 restart my-app || pm2 start app.js --name my-app
echo "Deployment completed!"
```
3. Закоммитьте и запушьте изменения.

#### Шаг 2. Настройка секретов в репозитории

1. В настройках репозитория перейдите в **Settings → Secrets and variables → Actions**.
2. Создайте следующие секреты:
   * `SERVER_IP` — IP‑адрес вашего VPS;
   * `SSH_USERNAME` — имя пользователя для SSH‑подключения;
   * `SSH_PRIVATE_KEY` — приватный SSH‑ключ (без пароля).

#### Шаг 3. Создание и настройка пайплайна

1. В репозитории создайте файл `.github/workflows/deploy.yml`.
2. Вставьте базовый шаблон пайплайна:
```yaml
name: Project CI/CD Pipeline

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
      - name: Install dependencies
        run: npm install
      - name: Run tests
        run: npm test

  deploy:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      - name: Setup SSH
        uses: webfactory/ssh-agent@v0.8.0
        with:
          ssh-private-key: ${{ secrets.SSH_PRIVATE_KEY }}
      - name: Add server to known hosts
        run: |
          mkdir -p ~/.ssh
          ssh-keyscan -H ${{ secrets.SERVER_IP }} >> ~/.ssh/known_hosts
      - name: Deploy to server
        run: |
          ssh ${{ secrets.SSH_USERNAME }}@${{ secrets.SERVER_IP }} "
            mkdir -p ~/apps/my-nodejs-app
            rsync -avz --delete ./ ${{ secrets.SSH_USERNAME }}@${{ secrets.SERVER_IP }}:~/apps/my-nodejs-app/
            cd ~/apps/my-nodejs-app
            chmod +x deploy.sh
            ./deploy.sh
          "
```

#### Шаг 4. Анализ и модификация переменных

1. **Найдите постоянные переменные** (встроенные в GitHub Actions):
   * выделите все упоминания `github.ref`, `github.repository` и т.д.;
   * объясните их назначение в комментариях к файлу (добавьте комментарии рядом с каждой переменной).
2. **Измените проект‑зависимые переменные:**
   * замените `node-version` на `20`;
   * измените ветку в `on:push:branches` с `main` на `production`;
   * обновите команду `run: npm test` на `npm run test:ci`, если такой скрипт есть в `package.json`.
3. **Адаптируйте под ОС:**
   * представьте, что сервер работает на Windows;
   * перепишите шаг `Deploy to server` для Windows (используйте PowerShell и `robocopy` вместо `rsync`);
   * укажите, какие секреты (`secrets`) останутся теми же, а какие потребуют изменений.
4. **Настройте инфраструктурно‑зависимые параметры:**
   * добавьте новый секрет `DEPLOY_PATH` в настройках репозитория (значение: `/var/www/myapp`);
   * используйте `${{ secrets.DEPLOY_PATH }}` вместо жёстко заданного пути `~/apps/my-nodejs-app` в шаге деплоя.

#### Шаг 5. Тестирование пайплайна

1. Закоммитьте и запушьте изменённый файл `deploy.yml` в ветку `main`.
2. Откройте вкладку **Actions** в репозитории и проследите за выполнением пайплайна.
3. Исправьте ошибки (если есть), обращая внимание на:
   * ошибки аутентификации (проверьте `SSH_PRIVATE_KEY`);
   * неверные пути (проверьте `DEPLOY_PATH`);
   * отсутствие команд на сервере (убедитесь, что `rsync` установлен).
4. Создайте Pull Request из новой ветки в `main` и убедитесь, что запускается только задание `test`.

#### Шаг 6. Дополнительные задания (по желанию)

1. Добавьте условие деплоя только для коммитов с сообщением, содержащим `[deploy]`:
   * используйте `github.event.head_commit.message` в условии `if`;
   * протестируйте, создав коммит с сообщением `Fix bug [deploy]`.

2. Модифицируйте пайплайн для работы с Docker:
   * добавьте шаг сборки Docker‑образа в задание `test`;
   * измените деплой, чтобы он запускал контейнер вместо копирования файлов.

### Контрольные вопросы

Ответьте на вопросы после выполнения работы:

1. Какие переменные в пайплайне являются постоянными (встроенными в GitHub)? Приведите 3 примера применения в конфигурациях и объясните их назначение.
2. Какие параметры пайплайна зависят от проекта? Приведите 3 примера.
3. Как изменится пайплайн, если сервер работает на Windows вместо Linux? Укажите 3 ключевых различия.
4. Почему секреты (`secrets`) следует хранить в настройках репозитория, а не в коде пайплайна? 
5. Что произойдёт, если удалить условие `if: github.ref == 'refs/heads/main'` в задании `deploy`? Как это повлияет на процесс деплоя?


**Итог:** после выполнения этого задания вы научитесь:
* различать постоянные и изменяемые переменные в GitHub Actions;
* настраивать пайплайн под конкретный проект и инфраструктуру;
* безопасно использовать секреты;
* адаптировать пайплайн под разные ОС;
* применять условную логику для управления выполнением заданий.
